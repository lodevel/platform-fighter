# Smash-Parity Plan

> Generated by a multi-agent spec-vs-code audit (2026-06): canonical Smash (Ultimate) behavior mapped against the codebase across **7 core systems + all 10 characters**, with **every gap independently fact-checked** by a second agent — both the "what Smash does" claim and the "what the code does" claim. Built to avoid invented frame data (see the honesty note below).

## Already shipped this session (do not redo)

Foundational fixes already committed on `main` (`f371763` … `98debf8`):
- **Jumpsquat / tap-jump buffer** — up+attack is no longer eaten by a jump (jumps used to be instant — no jumpsquat).
- **Up-smash & down-smash are now reachable** — added the vertical smash-flick; they were authored on every fighter but no input could fire them.
- **Down- and up-special read the stick** — down+special gave the cannon (neutral) before; up+special now also works via the up-stick on gamepad.
- **Up-tilt/up-smash reach in front; down-tilts reach the feet** — roster-wide hitbox reshape (regression-locked).
- **Nova down-special = real Samus morph-ball bomb** (timed fuse + bomb-jump self-bounce).
- **Pass-through-platform landing jitter** fixed; **Nova sprite feet** seated on the ground.

## Overview

**Spec target:** Modern *Super Smash Bros. Ultimate*-style baseline. Melee-era techs (wavedash, L-cancel, dash-dance depth) are explicitly **out of scope** — they are era-specific and not required for parity. The goal is to validate that all canonical Ultimate mechanics are wired end-to-end (input → move → hitbox → knockback → hitstun → getup), close the genuine structural gaps, and then run a balance/frame-data pass.

**How to read this plan:**
- Each gap line is a checkbox with the shape **canonical → current → action** and a severity tag.
- Severity: `[core]` = foundational mechanic / dispatch blocker, `[important]` = real feature gap or data-integrity issue, `[polish]` = documentation, dead-code, or feel/balance refinement.
- Items already verified as correct (e.g. per-fighter gravity, terminal velocity, short-hop, fast-fall, landing-lag auto-cancel, frame-data model, hitlag/hitstun formulas, shield system, deterministic physics) are **not relisted as gaps** — they are confirmed conformant and "maintain as-is."

**Honesty note — frame-data tuning:** This plan **does not fabricate** exact frame counts, knockback magnitudes, or KO percents. Where an item needs canonical numbers, it is flagged ⚠️ **needs frame-data source** and must be tuned against a real Ultimate frame-data reference (e.g. an Ultimate frame-data spreadsheet/wiki). The codebase deliberately deviates from exact Smash values in some places (notably `KNOCKBACK_PERCENT_TEMPER = 0.06`, which prevents the unbounded high-% explosion and tunes to ~75–80% KO thresholds) — these deviations are **intended** and KO percents will **not** match Ultimate exactly.

## Core mechanics — gap analysis & roadmap

### Extended-slot dispatch (directional grounded/aerial lights)
*The single biggest structural gap: a full data + resolver layer exists but the runtime never calls it.*

- [ ] **`[core]` Wire extended-slot routing into Character dispatch.** Canonical → direction-stick + attack on the ground should route neutral→jab, side→sideLight, up→upLight, down→downLight (with tilt fallback). Current → `extendedSlotResolver.ts` defines `resolveGroundedLightSlot()` / `resolveAerialLightSlot()` (exported from `src/characters/index.ts`), but `Character.ts` never calls them; it uses the legacy `groundedAttackInput.ts` dispatcher (jab/tilt/smash/utilt/dtilt/dashAttack/usmash/dsmash only — no `sideLightId` slot). Action → call `resolveGroundedLightSlot(moveset, direction)` from the grounded-attack dispatch (~`Character.ts:5863`) instead of the hard-coded slot table; this unblocks every per-character extended-move item below.
- [ ] **`[core]` Add `sideLightId` / `upLightId` / `downLightId` dispatch slots to Character.** Canonical → Character should expose a slot for every registered move type. Current → `Character.ts` private fields (`tiltAttackId`, `upTiltId`, `downTiltId`, `upSmashId`, `downSmashId`, `dashAttackSlotId`, …) have **no** `sideLightId`/`upLightId`/`downLightId`; `GroundedAttackSlots` interface in `groundedAttackInput.ts` also lacks them. Action → add optional `sideLightId?`/`upLightId?`/`downLightId?` fields to Character + `GroundedAttackSlots`; populate them on registration of extended-light moves. Dependency of the Wolf/Volt/Owl extended-move items.
- [ ] **`[important]` Add a generic `nextId?` (jab-chain) field to `AttackMove`.** Canonical → jab auto-chains jab1→jab2→jab3 on re-press. Current → works, but Wolf's chain is a Wolf-specific hardcoded `jabChain: { nextId }` data field (`Wolf.ts:200`), not a framework field on `AttackMove`. Action → document `nextId?` as an optional `AttackMove` field in `moveSchema.ts` (like `sweetSpot`); verify `groundedAttackInput.ts` respects the chain on re-press without per-fighter wiring; confirm all fighters populate it.

### Aerial-attack dispatch (up/down air in resolver)
- [ ] **`[core]` Route up/down stick to uair/dair in `moveResolver`.** Canonical → up-stick in air → uair, down-stick → dair, fallback to nair. Current → `moveResolver.ts:410-430` `aerialDirectionToSlot()` maps only `forward→fair`, `back→bair`, and defaults *everything else (incl. up/down)* to nair; lines 551-554 pass only `aerialNeutralId/aerialForwardId/aerialBackId` to `classifyAerialAttack` — never `aerialUpId/aerialDownId`. Action → (1) expand `AerialSlot` to include `'uair' | 'dair'` and map 'up'/'down' in `aerialDirectionToSlot()`; (2) pass `aerialUpId`/`aerialDownId` through; (3) add optional `uair`/`dair` to the moveset slots. Structural alignment, no tuning. *(Note: the live game does not currently call `resolveMoveFromInput` — this is mainly a data-integrity / test-path fix — but it must match runtime behavior.)*

### Ledge / tech / knockdown
- [ ] **`[core]` Spawn a real ledge-attack hitbox.** Canonical → ledge-attack is a frame-data move (startup/active/recovery) with a unique weak hitbox, released from edge-grab with directional input. Current → `ledgeHangState.ts` models the `'attack'` release; `Character.ts applyLedgeReleasePhysics()` (`7164-7171`) only applies an inward nudge — **no hitbox spawned** (comment at `7165` calls it a placeholder). Action → add a `LedgeAttackSchema` (hitbox/damage/knockback/frames, analogous to grab/throw), store on Character options, and on `ledgeReleased === 'attack'` fire it as a transient (like get-up attack); let each fighter author its own. ⚠️ **needs frame-data source** (damage/knockback/frames).
- [ ] **`[important]` Add wall-tech and ceiling-tech.** Canonical → a fighter tumbled into a wall or ceiling can tech on contact, gaining brief i-frames (wall → slide down; ceiling → fall). Current → **missing**: tumble/tech (`Character.ts:1906-1963`) only checks `isGrounded()` (a collision-count `> 0` test at `6725`); wall/ceiling contacts are never tracked, so horizontal/vertical launches pass through with no tech. Action → distinguish support/wall/ceiling contacts in the collision handler; gate tech on `isGrounded() || hitWall || hitCeiling`; implement wall-slide and ceiling-fall behaviors; grant `TECH_IFRAME_FRAMES` (20). Structural addition.
- [ ] **`[important]` Make get-up attack per-character authorable.** Canonical → get-up attack is a default knockdown wake-up option (low damage, low knockback, two-sided sweep); Ultimate adds variants. Current → `Character.ts startGetupAttack()` (`4519`) spawns a hardcoded transient (`GETUP_ATTACK_FRAMES=6`, `GETUP_ATTACK_DAMAGE=6`, fixed knockback `4,-3,0.12`) — **not** a named per-character move. Action → extract into an optional `GetUpAttackSchema` (damage/knockback/hitbox/animFrames) passed from Character options, used in `startGetupAttack()` instead of constants; fall back to defaults if omitted. ⚠️ **needs frame-data source**.
- [ ] **`[important]` Expose & validate tech-lockout frames.** Canonical → ~6–8 frame tech-buffer window; after it closes, re-press is locked out ~20–30 frames (anti-mash). Current → `TECH_WINDOW_FRAMES=8` (`Character.ts:567`) is correct; `techLockoutRemaining = TECH_LOCKOUT_FRAMES` is set at `1945` but `TECH_LOCKOUT_FRAMES` is **not defined/exported** (likely hardcoded in the tumble block). Action → find/expose `TECH_LOCKOUT_FRAMES` as a public constant near `TECH_WINDOW_FRAMES`, add canonical default if missing, document, and test that a mashed shield in tumble can't double-tech. ⚠️ **needs frame-data source** (~20–30).
- [ ] **`[important]` Validate knockdown getup-option frame counts & i-frames.** Canonical → prone ~26–30 frames; getup/roll/attack options have own i-frame budgets (~14–20); Ultimate getup-attack grants ~20 total i-frames covering attack + stand. Current → implemented: `KNOCKDOWN_FRAMES=26` (`571`), getup branch reads jump/attack/direction (`2034-2068`), `GETUP_IFRAME_FRAMES=14` (`573`). Action → verify whether 14 covers the *entire* getup process or only the attack-active window; if whole-process, raise toward ~20; tune getup-attack damage/range/direction; test okizeme. ⚠️ **needs frame-data source**.
- [ ] **`[important]` Tune tumble knockback threshold.** Canonical → tumble (tech-able) only above a knockback magnitude threshold (~7–8 units); weak pokes give non-tech-able knockback. Current → `TUMBLE_KNOCKBACK_THRESHOLD=7` (`561`); `tumbling = hypot(releaseVx,releaseVy) >= 7` (`1877`). Correct gate; validate the threshold against the weakest tumble-triggering / non-triggering moves. ⚠️ **needs frame-data source**.
- [ ] **`[important]` Confirm hitstun ≈ knockback magnitude scaling.** Canonical → hitstun ≈ 60–70% of knockback magnitude (frames), applied the frame after hitlag ends; strong hits stun longer to follow the arc. Current → implemented (`Character.ts applyHit ~6676-6680`, `combat.ts computeKnockback`; stored in `pendingKnockback.hitstunFrames`, applied at `1874`, drained at `1979`). Action → review the `computeKnockback` hitstun formula, confirm ~0.6–0.7× scaling, test jab@0% vs smash@50%, and that juggle hitstun matches fall-back-to-ground time. ⚠️ **needs frame-data source**.
- [ ] **`[important]` Tune ledge-roll i-frames & travel distance.** Canonical → ledge-roll i-frames during traversal (~24 frames), fixed distance (~1–1.5 body widths), vulnerable on the landing tail (readable). Current → implemented (`ledgeHangState.ts:702-739`, `rollIframes:24`, `rollDistance:96px`, `rollFrames:36`). Action → gather Ultimate ledge-roll data; adjust `rollDistance`/`rollIframes` so the i-frame tail ends before landing; test edge-hog reads. Balance-pass, not structural. ⚠️ **needs frame-data source**.

### Defensive / air options
- [ ] **`[important]` Decide air-dodge spec: directional vs neutral + landing lag.** Canonical → Ultimate has *directional* air dodge (stick gives a movement burst) and brief landing lag if grounded during dodge recovery. Current → `dodgeState.ts`: grounded dodge = spot/roll; airborne dodge is always neutral `'air'` (active 24 / iframe 20 / recovery 12), **no directional impulse** (`vx=0` at `Character.ts:2640`) and **no landing-lag** mechanism. Action → make a design decision: if matching Ultimate, add directional air-dodge impulse + landing-lag when `grounded` becomes true during `'air'` dodge recovery; if matching current intent (neutral only), document it as intentional. ⚠️ **needs frame-data source** if adding lag (~2–5 frames).

- [ ] **`[important]` Distinct run-stop / skid deceleration.** Canonical → releasing the stick out of a run gives a softer skid than a pivot (two different decel rates). - [ ] **`[polish]` Foxtrot.** Optional advanced tech (repeated dash flicks). Not named/optimized; approximable already. Add a dedicated state only if desired.
